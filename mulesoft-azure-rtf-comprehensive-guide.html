📋 Prerequisites & Setup Requirements

☁️ Azure Infrastructure Requirements

1

Azure Kubernetes Service (AKS) Cluster

A properly configured AKS cluster with Runtime Fabric installed. Minimum requirements:

  • Kubernetes version 1.24 or higher
  • Minimum 3 nodes with 4 vCPUs and 16GB RAM each
  • Premium storage class configured
  • Network connectivity to Anypoint Platform
2

Azure DevOps Project & Service Connections

Configure Azure DevOps with proper service connections:

  • Azure Resource Manager service connection
  • Kubernetes service connection to AKS cluster
  • Docker registry connection (if using custom images)
  • Proper RBAC permissions configured
3

Runtime Fabric Installation

Runtime Fabric must be installed and configured on AKS:

# Install Runtime Fabric using Helm helm repo add mulesoft https://repository-master.mulesoft.org/helm helm install rtf mulesoft/rtf \ --namespace rtf \ --set mule.licenseKeyBase64= \ --set cluster.provider=aks

🔒 Security Configuration & Best Practices

🚨 Critical Security Notice

All sensitive information must be stored in Azure Key Vault or Azure DevOps secure variables. Never commit credentials, tokens, or certificates to source control.

🔐 Credential Management

Azure Key Vault Integration

Store all sensitive credentials in Azure Key Vault:

# Key Vault secrets structure anypoint-username: $(kv-anypoint-username) anypoint-password: $(kv-anypoint-password) mule-license-key: $(kv-mule-license-base64) ssl-certificates: $(kv-ssl-cert-base64)
Configure Key Vault access:
  1. Create Azure Key Vault instance
  2. Add service principal access policy
  3. Configure Azure DevOps Key Vault task
  4. Reference secrets using $(variable-name) syntax
Secure Properties Configuration

Configure secure properties in your Mule application:

# secure-properties.yaml secure: properties: key: "${secure.key}" database: host: "secure::database.host" username: "secure::database.username" password: "secure::database.password"
Certificate Management

Manage SSL/TLS certificates securely:

  • Store certificates in Azure Key Vault as base64-encoded secrets
  • Use Azure certificate auto-rotation features
  • Implement certificate expiry monitoring
  • Configure proper cipher suites and protocols

🛡️ API Policy Management & Configuration

🚨 Security Policies

🔒
Client ID Enforcement
Required

Purpose: Ensures all API requests include valid client credentials for authentication.

# Client ID Enforcement Policy Configuration { "credentialsOriginHasHttpBasicAuthenticationHeader": "customExpression", "clientIdExpression": "#[attributes.headers['client_id']]", "clientSecretExpression": "#[attributes.headers['client_secret']]", "errorResponseOnInvalidClient": { "statusCode": 401, "reasonPhrase": "Unauthorized - Invalid Client Credentials" } }
Implementation: Applied automatically during the API Policy Management stage when applyPolicies parameter is set to true.
⏱️
Rate Limiting
Required

Purpose: Protects APIs from abuse by limiting request frequency per client.

# Rate Limiting Policy Configuration { "maximumRequests": 1000, "timePeriod": 60000, "timeUnit": "MILLISECONDS", "keySelector": "#[attributes.headers['client_id']]", "exposeHeaders": true, "errorResponse": { "statusCode": 429, "reasonPhrase": "Too Many Requests" } }
⚠️ Rate Limit Guidelines:
  • Development: 100 requests/minute
  • Testing: 500 requests/minute
  • Production: 1000+ requests/minute
🌐
CORS Policy
Optional

Purpose: Enables cross-origin requests for web applications.

# CORS Policy Configuration { "allowedOrigins": ["https://mycompany.com", "https://app.mycompany.com"], "allowedMethods": ["GET", "POST", "PUT", "DELETE", "OPTIONS"], "allowedHeaders": ["Content-Type", "Authorization", "X-Requested-With"], "maxAge": 3600, "allowCredentials": true }

IP Whitelisting
Security

Purpose: Restricts API access to specific IP addresses or ranges.

# IP Whitelisting Policy { "ipList": [ "192.168.1.0/24", "10.0.0.0/16", "172.16.0.0/12" ], "errorResponse": { "statusCode": 403, "reasonPhrase": "Forbidden - IP not whitelisted" } }
🔐
OAuth 2.0 Token Validation
Security

Purpose: Validates OAuth 2.0 access tokens for secure API access.

# OAuth 2.0 Token Validation Policy { "authorizationServerUrl": "https://oauth-server.company.com", "introspectionUrl": "https://oauth-server.company.com/introspect", "clientId": "${oauth.client.id}", "clientSecret": "${oauth.client.secret}", "scopes": ["read", "write", "admin"] }
📝
Message Logging
Optional

Purpose: Logs API requests and responses for audit and debugging.

# Message Logging Policy { "logLevel": "INFO", "logRequestHeaders": true, "logResponseHeaders": false, "logRequestBody": false, "logResponseBody": false, "maxPayloadSize": 1024, "category": "com.mulesoft.policy.message-logging" }
⚠️ Privacy Notice: Be cautious when logging request/response bodies as they may contain sensitive data. Always comply with privacy regulations.

⚙️ Policy Application Process

Automatic Policy Application

The pipeline automatically applies policies when applyPolicies parameter is enabled:

# Pipeline applies policies in this order: 1. Client ID Enforcement (Authentication) 2. Rate Limiting (Traffic Control) 3. CORS (Cross-Origin Access) 4. IP Whitelisting (Network Security) 5. Message Logging (Audit Trail)
Custom Policy Configuration

To add custom policies, modify the policy management stage:

# Add custom policy in pipeline anypoint-cli api-mgr policy apply \ --apiId "$API_ID" \ --policyId custom-security-policy \ --configurationData '{ "customParameter": "value", "securityLevel": "high" }'
Policy Validation & Testing

After policy application, the pipeline validates policy effectiveness:

  • ✅ Policy application status verification
  • ✅ API endpoint accessibility testing
  • ✅ Policy configuration validation
  • ✅ Performance impact assessment

🚀 Deployment Strategies & Configuration

📦 Deployment Strategies

📦
Standard Deployment
Default

Use Case: Regular deployments with minimal complexity.

  • ✅ Simple deployment process
  • ✅ Suitable for development and testing
  • ⚠️ Brief downtime during deployment
# Standard Deployment Configuration replicas: 2 strategy: Recreate maxUnavailable: 50% maxSurge: 0
🔵
Blue-Green Deployment
Production

Use Case: Zero-downtime deployments with instant rollback capability.

  • ✅ Zero downtime deployment
  • ✅ Instant rollback capability
  • ✅ Full environment isolation
  • ⚠️ Requires double resources temporarily
# Blue-Green Deployment Process 1. Deploy new version to "Blue" environment 2. Run health checks and validation 3. Switch traffic from "Green" to "Blue" 4. Monitor for issues 5. Decommission old "Green" environment
🐤
Canary Deployment
Advanced

Use Case: Gradual rollout to subset of users for risk mitigation.

  • ✅ Gradual traffic shifting
  • ✅ Risk mitigation
  • ✅ A/B testing capabilities
  • ⚠️ Complex monitoring required
# Canary Deployment Configuration initial_traffic: 10% increment_step: 25% success_criteria: - error_rate < 1% - response_time < 500ms - success_rate > 99%
🔄
Rolling Update
Balanced

Use Case: Gradual replacement of instances with minimal resource usage.

  • ✅ Gradual instance replacement
  • ✅ Resource efficient
  • ✅ Configurable update pace
  • ⚠️ Temporary mixed versions
# Rolling Update Configuration maxUnavailable: 25% maxSurge: 25% progressDeadlineSeconds: 600 revisionHistoryLimit: 3
