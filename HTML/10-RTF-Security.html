🛡️ Security Overview

Self-managed Runtime Fabric deployments require comprehensive security configurations to protect against threats, ensure data integrity, and maintain compliance standards. This guide covers all security aspects from infrastructure to application-level protection.

🔒 Infrastructure Security Critical

  • Network segmentation and isolation
  • Firewall configuration and rules
  • VPN/Private network access
  • OS-level hardening

🔑 Identity & Access Management Critical

  • RBAC implementation
  • Multi-factor authentication
  • Service account management
  • API key rotation

🛡️ Data Protection High

  • Encryption at rest and in transit
  • TLS/SSL configuration
  • Secret management
  • Key management systems

📊 Monitoring & Auditing High

  • Security event logging
  • Intrusion detection
  • Compliance monitoring
  • Threat intelligence

🔐 Authentication & Authorization

🎯 RBAC Implementation

# Create RTF security namespace kubectl create namespace rtf-security # Create service accounts with minimal permissions cat << EOF | kubectl apply -f - apiVersion: v1 kind: ServiceAccount metadata: name: rtf-operator namespace: rtf-system --- apiVersion: v1 kind: ServiceAccount metadata: name: rtf-monitoring namespace: rtf-system --- apiVersion: v1 kind: ServiceAccount metadata: name: rtf-admin namespace: rtf-system EOF # Create RBAC policies for RTF operations cat << EOF | kubectl apply -f - apiVersion: rbac.authorization.k8s.io/v1 kind: ClusterRole metadata: name: rtf-operator-role rules: - apiGroups: ["apps"] resources: ["deployments", "replicasets"] verbs: ["get", "list", "watch", "create", "update", "patch"] - apiGroups: [""] resources: ["pods", "services", "configmaps", "secrets"] verbs: ["get", "list", "watch", "create", "update", "patch"] - apiGroups: ["networking.k8s.io"] resources: ["ingresses", "networkpolicies"] verbs: ["get", "list", "watch", "create", "update", "patch"] --- apiVersion: rbac.authorization.k8s.io/v1 kind: ClusterRoleBinding metadata: name: rtf-operator-binding roleRef: apiGroup: rbac.authorization.k8s.io kind: ClusterRole name: rtf-operator-role subjects: - kind: ServiceAccount name: rtf-operator namespace: rtf-system EOF

🛡️ Network Security

🔥 Firewall Configuration

🚨 Critical Security Requirement: Implement strict firewall rules to control access to RTF components. Only allow necessary traffic and block all other connections.
# Create NetworkPolicy to restrict pod-to-pod communication cat << EOF | kubectl apply -f - apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: rtf-network-policy namespace: rtf-system spec: podSelector: {} policyTypes: - Ingress - Egress ingress: - from: - namespaceSelector: matchLabels: name: rtf-system - namespaceSelector: matchLabels: name: ingress-nginx ports: - protocol: TCP port: 8080 - protocol: TCP port: 8443 egress: - to: [] ports: - protocol: TCP port: 443 # HTTPS outbound - protocol: TCP port: 53 # DNS - protocol: UDP port: 53 # DNS EOF # Configure Kubernetes native firewall rules cat << EOF | kubectl apply -f - apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: deny-all-ingress namespace: rtf-system spec: podSelector: {} policyTypes: - Ingress --- apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: allow-rtf-internal namespace: rtf-system spec: podSelector: matchLabels: app: runtime-fabric policyTypes: - Ingress ingress: - from: - podSelector: matchLabels: app: runtime-fabric ports: - protocol: TCP port: 8080 EOF

🌐 VPN and Private Network Setup

# Install WireGuard VPN for secure access # For Ubuntu/Debian systems sudo apt update && sudo apt install -y wireguard # Generate server keys wg genkey | tee /etc/wireguard/server_private.key | wg pubkey > /etc/wireguard/server_public.key # Create WireGuard server configuration cat << EOF > /etc/wireguard/wg0.conf [Interface] Address = 10.100.0.1/24 ListenPort = 51820 PrivateKey = $(cat /etc/wireguard/server_private.key) PostUp = iptables -A FORWARD -i %i -j ACCEPT; iptables -A FORWARD -o %i -j ACCEPT; iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE PostDown = iptables -D FORWARD -i %i -j ACCEPT; iptables -D FORWARD -o %i -j ACCEPT; iptables -t nat -D POSTROUTING -o eth0 -j MASQUERADE # Client configurations will be added here [Peer] PublicKey = CLIENT_PUBLIC_KEY AllowedIPs = 10.100.0.2/32 EOF # Enable IP forwarding echo 'net.ipv4.ip_forward=1' >> /etc/sysctl.conf sysctl -p # Start WireGuard service systemctl enable wg-quick@wg0 systemctl start wg-quick@wg0

🔒 TLS/SSL Configuration

# Create comprehensive TLS configuration for RTF cat << EOF | kubectl apply -f - apiVersion: v1 kind: Secret metadata: name: rtf-tls-certs namespace: rtf-system type: kubernetes.io/tls data: tls.crt: $(cat your-certificate.crt | base64 -w 0) tls.key: $(cat your-private.key | base64 -w 0) --- apiVersion: v1 kind: ConfigMap metadata: name: rtf-tls-config namespace: rtf-system data: tls.conf: | ssl_protocols TLSv1.2 TLSv1.3; ssl_ciphers ECDHE-RSA-AES256-GCM-SHA512:DHE-RSA-AES256-GCM-SHA512:ECDHE-RSA-AES256-GCM-SHA384:DHE-RSA-AES256-GCM-SHA384; ssl_ecdh_curve secp384r1; ssl_session_timeout 10m; ssl_session_cache shared:SSL:10m; ssl_session_tickets off; ssl_stapling on; ssl_stapling_verify on; add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload"; add_header X-Content-Type-Options nosniff; add_header X-Frame-Options DENY; add_header X-XSS-Protection "1; mode=block"; EOF

🔑 Secret Management

🗝️ HashiCorp Vault Integration

💡 Best Practice: Use HashiCorp Vault for centralized secret management, automatic rotation, and audit trails.
# Install HashiCorp Vault using Helm helm repo add hashicorp https://helm.releases.hashicorp.com helm repo update # Deploy Vault with high availability helm install vault hashicorp/vault \ --namespace rtf-security \ --create-namespace \ --set server.ha.enabled=true \ --set server.ha.replicas=3 \ --set server.dataStorage.enabled=true \ --set server.dataStorage.size=10Gi \ --set ui.enabled=true \ --set ui.serviceType=LoadBalancer # Initialize Vault (run this once) kubectl exec -ti vault-0 -n rtf-security -- vault operator init # Unseal Vault on all replicas kubectl exec -ti vault-0 -n rtf-security -- vault operator unseal UNSEAL_KEY_1 kubectl exec -ti vault-0 -n rtf-security -- vault operator unseal UNSEAL_KEY_2 kubectl exec -ti vault-0 -n rtf-security -- vault operator unseal UNSEAL_KEY_3 # Configure Kubernetes authentication kubectl exec -ti vault-0 -n rtf-security -- sh vault auth enable kubernetes vault write auth/kubernetes/config \ token_reviewer_jwt="$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)" \ kubernetes_host="https://$KUBERNETES_PORT_443_TCP_ADDR:443" \ kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt

🔐 Secret Rotation Policy

# Create secret rotation policy for RTF components cat << EOF | kubectl apply -f - apiVersion: v1 kind: ConfigMap metadata: name: secret-rotation-policy namespace: rtf-system data: policy.json: | { "rotation_policies": { "database_passwords": { "rotation_interval": "30d", "backup_count": 3, "notification_before": "7d" }, "api_keys": { "rotation_interval": "90d", "backup_count": 5, "notification_before": "14d" }, "tls_certificates": { "rotation_interval": "365d", "backup_count": 2, "notification_before": "30d" }, "jwt_secrets": { "rotation_interval": "60d", "backup_count": 3, "notification_before": "7d" } } } EOF # Create CronJob for automatic secret rotation cat << EOF | kubectl apply -f - apiVersion: batch/v1 kind: CronJob metadata: name: secret-rotation namespace: rtf-system spec: schedule: "0 2 * * 0" # Weekly on Sunday at 2 AM jobTemplate: spec: template: spec: containers: - name: secret-rotator image: vault:1.15.0 command: - /bin/sh - -c - | # Rotate secrets based on policy vault kv put secret/rtf/db-password password=$(openssl rand -base64 32) vault kv put secret/rtf/api-key key=$(uuidgen) # Add more rotation logic here env: - name: VAULT_ADDR value: "http://vault.rtf-security.svc.cluster.local:8200" - name: VAULT_TOKEN valueFrom: secretKeyRef: name: vault-token key: token restartPolicy: OnFailure EOF
